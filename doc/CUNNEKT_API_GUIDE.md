# Cunnekt WhatsApp API Integration Guide

> **Prepared:** 20 July 2026  
> **Purpose:** Implementation reference for WordPress, Forminator, MERN/Node.js, and other server-side integrations.

## 1. Important: Cunnekt currently exposes two API formats

The linked Postman documentation is labelled by Cunnekt as its **Legacy API**. Cunnekt's newer documentation uses a different endpoint and a Meta/WhatsApp Cloud API-style payload.

Do not combine the endpoint from one version with the payload from the other.

| API generation | Template endpoint | Recipient field | Template field |
|---|---|---|---|
| Legacy Postman API | `POST https://app.cunnekt.com/api/v1/whatsapp/sendtemplate` | `mobile` | `templateid` |
| Current REST API | `POST https://app.cunnekt.com/restapi/v1/whatsapp/sendtemplate` | `to` | `template.name` |

### Recommended selection

1. Open the Cunnekt dashboard.
2. Go to the approved template list.
3. Open the template's **API Payload** action.
4. Use the endpoint and JSON payload shown by your own dashboard.
5. Do not transform that payload into another Cunnekt API format unless Cunnekt support instructs you to do so.

For a new integration, prefer the current REST API when it is enabled for the account. Keep the legacy adapter available if the account/dashboard still generates the legacy payload.

## 3. Authentication

Every request must contain the Cunnekt API key in the HTTP header.

```http
API-KEY: your_api_key
Content-Type: application/json
```

### Generate the key

In the Cunnekt dashboard:

1. Open **Settings**.
2. Open **API & Webhooks**.
3. Select **Generate API Key** when a key does not already exist.
4. Copy and store it securely.

Cunnekt's current documentation says each workspace/channel has its own API key.

### Never expose the key

Do not put the API key in:

- Browser JavaScript
- Elementor HTML widgets
- Public Git repositories
- Mobile applications
- Client-visible AJAX payloads
- WordPress options rendered into page source

Use it only from a trusted server-side application.

---

## 4. Phone-number format

Use the recipient's full international number:

```text
Country code + national mobile number
```

Example for India:

```text
919876543210
```

Recommended normalization rules:

- Remove spaces, brackets, and hyphens.
- Remove the leading `+` unless the account-generated payload explicitly contains it.
- Validate the country code.
- Do not assume every ten-digit number belongs to India.
- Store the normalized E.164-style value separately from the display value.

# Part A — Legacy Postman API

## 5. Legacy send-template endpoint

```http
POST https://app.cunnekt.com/api/v1/whatsapp/sendtemplate
```

### Headers

```http
API-KEY: your_api_key
Content-Type: application/json
```

### Core legacy payload

The legacy documentation identifies these top-level fields:

```json
{
  "mobile": "919876543210",
  "templateid": "YOUR_TEMPLATE_ID",
  "overridebot": "no",
  "template": {
    "components": []
  }
}
```

| Field | Required | Description |
|---|---:|---|
| `mobile` | Yes | Receiver's mobile number with country code |
| `templateid` | Yes | Cunnekt/WhatsApp template ID shown in the template list/API payload |
| `overridebot` | Account/payload dependent | Whether the outgoing message should override bot behaviour; use exactly what the generated payload specifies |
| `template.components` | Template dependent | Header, body, button, carousel, or other template parameters |

> **Do not invent the template ID.** Retrieve it from the Cunnekt template list or the template's generated API payload.

## 6. Legacy text-variable example

Use this structure only when it matches the API payload generated by your Cunnekt dashboard.

```json
{
  "mobile": "919876543210",
  "templateid": "YOUR_TEMPLATE_ID",
  "overridebot": "no",
  "template": {
    "components": [
      {
        "type": "body",
        "parameters": [
          {
            "type": "text",
            "text": "Aarav Sharma"
          },
          {
            "type": "text",
            "text": "International Scratch Olympiad 2026"
          }
        ]
      }
    ]
  }
}
```

The order of `parameters` must match the order of variables in the approved template.

For example:

```text
Hello {{1}}, your registration for {{2}} is successful.
```

maps to:

```text
{{1}} = Aarav Sharma
{{2}} = International Scratch Olympiad 2026
```

```json
    "components": [
      {
        "type": "header",

          {
            "type": "image",
            "image": {
              "link": "https://example.com/public/banner.jpg"
            }
          }
        ]
      },
      {
        "type": "body",
        "parameters": [
          {
            "type": "text",

          },
```

Media requirements:

- The URL must be publicly reachable over HTTPS.
- It must return the actual media file, not an HTML login page.
- Avoid expiring URLs unless their lifetime safely exceeds Cunnekt's fetch time.
- Match the approved header type: image, video, or document.

# Part B — Current Cunnekt REST API

## 10. Current send-template endpoint

```http
POST https://app.cunnekt.com/restapi/v1/whatsapp/sendtemplate
```

> The current documentation's endpoint section displays `GET`, but its cURL example uses `POST`, and sending a message requires a request body. Use `POST` unless the account-specific Cunnekt payload explicitly says otherwise.

### Headers

```http
API-KEY: your_api_key
Content-Type: application/json
```

## 11. Current REST payload

```json
{
  "messaging_product": "whatsapp",
  "recipient_type": "individual",
  "to": "919876543210",
  "type": "template",
  "template": {
    "name": "registration_success",
    "language": {
      "code": "en"
    },
    "components": [
      {
        "type": "body",
        "parameters": [
          {
            "type": "text",
```

| Field | Required | Description |
|---|---:|---|
| `messaging_product` | Yes | Use `whatsapp` |
| `recipient_type` | Yes | Usually `individual` |

## 13. Template management API

### Retrieve templates

```http
GET https://app.cunnekt.com/restapi/v1/whatsapp/templates
```

### Create a template

```http
POST https://app.cunnekt.com/restapi/v1/whatsapp/templates
```

### Create-template payload skeleton

```json
{
  "name": "registration_success",
  "category": "UTILITY",
  "language": "en",
  "parameter_format": "POSITIONAL",
  "components": []
}
```

The complete `components` structure is defined by the WhatsApp template type. Template creation is normally easier through the Cunnekt dashboard unless the integration must programmatically manage templates.

### Template naming guidance

- Use lowercase names.
- Use underscores instead of spaces.
- Keep one purpose per template.
- Keep variable order stable.
- Do not repurpose an approved utility template for marketing content.

Examples:

```text
gema_registration_success
scratch_olympiad_registration
certificate_delivery
payment_confirmation
event_reminder
otp_verification
```

## 14. WhatsApp conversation rules

Cunnekt's current documentation describes four conversation categories:

- **Service:** User-initiated conversation.
- **Marketing:** Promotions, offers, announcements, and similar business-initiated content.
- **Utility:** Transactional updates and reminders.
- **Authentication:** OTP and identity-verification messages.

When a user messages the business, a 24-hour service window begins from the user's latest message.

During an active service window:

- Free-form text, media, and interactive messages may be sent.
- A template is not required.

Outside the service window:

- An approved template is required.
- Free-form outbound messages are not allowed.

For registration confirmations, payment updates, certificates, and event reminders, a utility template is usually appropriate. OTP messages must use an authentication template.

# Part C — Webhooks

## 15. Configure a webhook

In the Cunnekt dashboard:

1. Go to **Settings**.
2. Open **API & Webhooks**.
3. Enter a publicly accessible HTTPS endpoint.
4. Save the configuration.

Example:

```text
https://example.com/wp-json/forminator-cunnekt/v1/webhook
```

or:

```text
https://api.example.com/api/webhooks/cunnekt
```

## 16. Webhook event categories

Cunnekt documents two broad webhook categories.

### Channel webhooks

WhatsApp-specific events such as:

- Incoming messages
- Sent/delivered/read/failed status updates
- Template approval status updates

### Platform webhooks

Cunnekt-platform events such as:

- Conversation state changes
- Contact creation or updates
- Campaign lifecycle events
- Automation and assignment changes

## 17. Webhook processing rules

A webhook endpoint should:

1. Accept an HTTP `POST` request.
2. Validate the basic payload structure.
3. Persist or enqueue the raw payload.
4. Return `200 OK` in under one second.
5. Process business logic asynchronously.
6. Be idempotent because duplicate deliveries are possible.

Do not perform slow operations before acknowledging the request, including:

- Generating PDFs
- Sending emails
- Calling several external APIs
- Long database transactions
- Rebuilding reports

## 18. Generic webhook handler pseudocode

```php
function handle_cunnekt_webhook(WP_REST_Request $request): WP_REST_Response {
    $payload = $request->get_json_params();

    if (!is_array($payload)) {
        return new WP_REST_Response(['ok' => false], 400);
    }

    // Store the raw payload or enqueue it for asynchronous processing.
    cunnekt_enqueue_webhook($payload);

    // Acknowledge immediately.
    return new WP_REST_Response(['ok' => true], 200);
}
```

Because Cunnekt's publicly accessible webhook pages do not currently provide a complete, stable payload schema, store the raw event and build parsers from real sandbox payloads captured from your account.

# Part D — WordPress implementation

## 19. Store credentials in `wp-config.php`

```php
define('CUNNEKT_API_KEY', 'replace-with-real-key');
define('CUNNEKT_API_VERSION', 'legacy'); // legacy or rest
```

For different WordPress websites using one Cunnekt account, the same API key can technically be configured server-side when Cunnekt permits it. A stronger design uses a central gateway so the key exists in only one system.

## 20. WordPress legacy send function

```php
<?php
/**
 * Send a Cunnekt legacy template message.
 *
 * @return array{success:bool,status:int,body:array|string,error:?string}
 */
function gema_cunnekt_send_legacy_template(
    string $mobile,
    string $template_id,
    array $components = [],
    string $override_bot = 'no'
): array {
    if (!defined('CUNNEKT_API_KEY') || CUNNEKT_API_KEY === '') {
        return [
            'success' => false,
            'status'  => 0,
            'body'    => '',
            'error'   => 'Cunnekt API key is not configured.',
        ];
    }

    $mobile = preg_replace('/\D+/', '', $mobile) ?? '';

    if ($mobile === '' || $template_id === '') {
        return [
            'success' => false,
            'status'  => 0,
            'body'    => '',
            'error'   => 'Mobile number and template ID are required.',
        ];
    }

    $payload = [
        'mobile'      => $mobile,
        'templateid'  => $template_id,
        'overridebot' => $override_bot,
        'template'    => [
            'components' => array_values($components),
        ],
    ];

    $response = wp_remote_post(
        'https://app.cunnekt.com/api/v1/whatsapp/sendtemplate',
        [
            'timeout' => 20,
            'headers' => [
                'API-KEY'     => CUNNEKT_API_KEY,
                'Content-Type' => 'application/json',
                'Accept'       => 'application/json',
            ],
            'body' => wp_json_encode($payload),
        ]
    );

    if (is_wp_error($response)) {
        return [
            'success' => false,
            'status'  => 0,
            'body'    => '',
            'error'   => $response->get_error_message(),
        ];
    }

    $status   = wp_remote_retrieve_response_code($response);
    $raw_body = wp_remote_retrieve_body($response);
    $decoded  = json_decode($raw_body, true);

    return [
        'success' => $status >= 200 && $status < 300,
        'status'  => $status,
        'body'    => is_array($decoded) ? $decoded : $raw_body,
        'error'   => $status >= 200 && $status < 300 ? null : 'Cunnekt request failed.',
    ];
}
```

## 22. Forminator integration flow

Recommended plugin workflow:

```text
Forminator form submitted
        ↓
Validate consent and phone number
        ↓
Map Form ID to template configuration
        ↓
Create message job in custom queue table
        ↓
Return form success immediately
        ↓
Background worker sends through Cunnekt
        ↓
Store HTTP status + raw API response
        ↓
Retry transient failures
        ↓
Update status from Cunnekt webhook
```

Recommended mapping record:

```json
{
  "form_id": 123,
  "enabled": true,
  "api_version": "legacy",

    "select-1"
  ],

  "send_to_admin": true
}
```

# Part E — Node.js implementation

## 23. Node.js current REST example

```js
const CUNNEKT_ENDPOINT =
  'https://app.cunnekt.com/restapi/v1/whatsapp/sendtemplate';

/**
 * @param {{
 *   to: string,
 *   templateName: string,
 *   languageCode?: string,
 *   components?: unknown[]
 * }} input
 */
export async function sendCunnektTemplate(input) {
  const apiKey = process.env.CUNNEKT_API_KEY;

  if (!apiKey) {
    throw new Error('CUNNEKT_API_KEY is not configured');
  }

  const to = input.to.replace(/\D/g, '');

  if (!to || !input.templateName) {
    throw new Error('Recipient and template name are required');
  }

  const response = await fetch(CUNNEKT_ENDPOINT, {
    method: 'POST',
    headers: {
      'API-KEY': apiKey,
      'Content-Type': 'application/json',
      Accept: 'application/json',
    },
    body: JSON.stringify({
      messaging_product: 'whatsapp',
      recipient_type: 'individual',
      to,
      type: 'template',
      template: {
        name: input.templateName,
        language: { code: input.languageCode ?? 'en' },
        components: input.components ?? [],
      },
    }),
    signal: AbortSignal.timeout(20_000),
  });

  const rawBody = await response.text();
  let body;

  try {
    body = JSON.parse(rawBody);
  } catch {
    body = rawBody;
  }

  if (!response.ok) {
    const error = new Error(`Cunnekt returned HTTP ${response.status}`);
    error.status = response.status;
    error.responseBody = body;
    throw error;
  }

  return {
    status: response.status,
    body,
  };
}
```

# Part F — Reliability and security

## 24. Response handling

Cunnekt's public documentation does not currently define one stable response schema across both API generations.

Therefore:

- Treat any HTTP `2xx` response as accepted by the Cunnekt API, not necessarily delivered to WhatsApp.
- Save the complete raw response.
- Extract a message ID only after confirming the actual response field from your account.
- Use delivery webhooks as the source of truth for delivered/read/failed status.
- Do not hard-code a guessed field such as `messageId`, `id`, or `messages[0].id` without a captured response.

Recommended database columns:

```text
provider
api_version
website_id
form_id
recipient
normalized_recipient
template_reference
request_payload_redacted
http_status
provider_message_id
provider_response
status
attempt_count
next_attempt_at
last_error
created_at
updated_at
```

Redact the API key and sensitive personal information before logging.

## 25. Retry policy

Retry only failures likely to be temporary:

- Network timeout
- Connection failure
- HTTP `408`
- HTTP `429`
- HTTP `500`, `502`, `503`, or `504`

Suggested schedule:

```text
Attempt 1: immediately
Attempt 2: after 1 minute
Attempt 3: after 5 minutes
Attempt 4: after 15 minutes
Attempt 5: after 1 hour
```

Do not automatically retry permanent validation failures such as malformed payloads, invalid template configuration, or invalid phone numbers until the data is corrected.

## 26. Idempotency

A form submission can be delivered more than once because of browser retries, plugin hooks, queue retries, or webhook duplicates.

Create an internal idempotency key, for example:

```text
SHA-256(website_id + form_id + submission_id + recipient + template_reference)
```

Before enqueueing a send, check whether that key already has a successful or active job.

## 27. Consent

Before sending business-initiated WhatsApp messages:

- Record the user's WhatsApp consent.
- Store consent timestamp and source.
- State what type of messages will be sent.
- Provide an opt-out mechanism where required.
- Do not use registration consent as blanket permission for unrelated marketing.

## 28. Multiple websites

For several websites under the same organization:

### Acceptable small setup

```text
Website A ─┐
Website B ─┼── Cunnekt API
Website C ─┘
```

Each website keeps the key server-side and uses its own template mapping and logs.

### Recommended scalable setup

```text
Website A ─┐
Website B ─┤
Website C ─┼── Central WhatsApp Gateway ── Cunnekt API
Website N ─┘
```

Benefits of a central gateway:

- The Cunnekt key is stored in one place.
- Per-website authentication and rate limits are possible.
- Centralized logging and retries.
- One website can be disabled without rotating credentials everywhere.
- Template IDs/names can be mapped centrally.
- Easier migration between legacy and current Cunnekt APIs.

Use separate sender numbers or workspaces for unrelated brands when the WhatsApp display name would otherwise misrepresent the sender.

# Part G — Configuration recommendation

## 29. Provider configuration

A reusable `CunnektWhatsAppProvider` should not guess the API format. Use explicit configuration:

```json
{
  "provider": "cunnekt",
  "apiVersion": "legacy",
  "baseUrl": "https://app.cunnekt.com",
  "sendTemplatePath": "/api/v1/whatsapp/sendtemplate",
  "authHeader": "API-KEY",
  "timeoutMs": 20000,
  "maxAttempts": 5,
  "responseIdPaths": []
}
```

Current REST configuration:

```json
{
  "provider": "cunnekt",
  "apiVersion": "rest-v1",
  "baseUrl": "https://app.cunnekt.com",
  "sendTemplatePath": "/restapi/v1/whatsapp/sendtemplate",
  "authHeader": "API-KEY",
  "timeoutMs": 20000,
  "maxAttempts": 5,
  "responseIdPaths": []
}
```

After the first successful sandbox request, add only the verified provider-message-ID path.

## 30. Environment variables

```dotenv
CUNNEKT_API_KEY=
CUNNEKT_API_VERSION=legacy
CUNNEKT_BASE_URL=https://app.cunnekt.com
CUNNEKT_TIMEOUT_MS=20000
```

The current public documentation does not specify a webhook-signature header or verification algorithm. Do not invent one. Ask Cunnekt support whether the account supports a webhook secret/signature. Until confirmed, protect the endpoint with HTTPS, rate limiting, strict payload limits, logging, and optional secret URL/header controls agreed with Cunnekt.
